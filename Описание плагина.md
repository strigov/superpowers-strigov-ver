# superpowers-strigov-ver — описание плагина

Плагин для Claude Code, который заменяет режим «один Claude делает всё» на **структурированный многомодельный рабочий процесс**: Sonnet оркестрирует, Opus принимает решения, Codex реализует и проверяет.

## Идея

По умолчанию Claude Code использует одну модель для всего — планирование, написание кода, ревью, коммит. На небольших задачах это работает, но на чём-то нетривиальном начинаются проблемы: модель, которая писала код, сама же его проверяет (конфликт интересов); нет плана, с которым можно сверять результат; каждая сессия начинается с нуля.

Плагин заменяет это разделением труда, соответствующим тому, в чём каждая модель реально сильна:

- **Sonnet** (главный поток) — дешёвая быстрая оркестрация. Читает вывод, диспатчит следующий шаг, делает git. Никогда не пишет продовый код.
- **Opus** (субагент) — суждение. Пишет план, ревьюит код, диагностирует блокеры. Каждый раз получает чистый контекст.
- **Codex xhigh** — скептичное read-only ревью. Проверяет план до начала реализации; делает контрольное ревью кода после того, как Opus его одобрил.
- **Codex high** — реализация. Пишет файлы.

В результате: план существует до того, как написана первая строка кода; два независимых ревьюера подписываются перед коммитом; сессия возобновляема через дни.

## Рабочий процесс

### Триаж

Каждая задача сначала проходит триаж:

```
Изменение одного файла, < 20 строк, без архитектурного суждения?
  ДА  → Sonnet quickfix субагент (быстрый путь, без цикла ревью)
  НЕТ → Полный протокол
```

### Полный протокол (5 шагов)

```
Шаг 1  Opus субагент пишет план
       └─ docs/plans/YYYY-MM-DD-<slug>.md
          YAML frontmatter: фазы, статусы
          Разделы: Цель · Файлы · Контракты · Стратегия тестирования · Риски · Фазы

Шаг 2  Codex xhigh ревьюит план  (цикл, лимит=4)
       └─ APPROVED → коммит плана, переход к шагу 3
          CHANGES_REQUESTED → Opus правит план → Codex проверяет снова
          4 раунда без одобрения → эскалация к пользователю

Шаг 3  Codex high реализует текущую фазу  (--write --effort high)
       └─ DONE → шаг 4
          DONE_WITH_CONCERNS → Sonnet читает замечания, решает
          NEEDS_CONTEXT → предоставить контекст, повторно диспатчить
          BLOCKED → Opus диагностирует → эскалация или добавить контекст и повторить

Шаг 4  Слитое ревью  (цикл, лимит=4)
       ├─ 4.1  Opus ревьюит: сначала соответствие плану, потом качество
       │        REVIEW_OK → переход к 4.2
       │        REVIEW_BLOCKING → назад к Codex high со списком правок
       └─ 4.2  Codex xhigh контрольное ревью (только если 4.1 прошло)
                Ортогональный угол: граничные случаи, гонки, безопасность, расхождения плана и кода
                ОБА ЧИСТЫ → авто-коммит, переход к следующей фазе
                BLOCKING → назад к Codex high с объединённым списком

Шаг 5  Авто-коммит
       └─ Обновляет frontmatter плана (статус фазы: done → следующая: in-progress)
          Conventional-commit: краткий заголовок, тело про "зачем"
          Трейлер Co-Authored-By
          Никогда git push без явного согласия пользователя
```

### Многофазные планы

В плане может быть несколько фаз (Ф1, Ф2, Ф3…). Шаги 1 и 2 покрывают **все фазы сразу** заранее. Шаги 3–5 выполняются **по одной фазе** без лишних вопросов между ними — план уже согласован.

### Возобновление между сессиями

Файл плана — единственный источник истины. Если сессия прервана, оркестратор сканирует `docs/plans/` в поисках плана со `status: in-progress`, показывает его пользователю и продолжает с нужной фазы.

## Защитные механизмы

**Anti-pingpong**: если Codex поднимает замечание, которое уже было явно отклонено с аргументацией в предыдущем раунде, оно помечается `resolved-by-decision` и больше не зацикливается.

**Детектор отсутствия прогресса**: если два подряд раунда ревью дают идентичный список блокеров — цикл немедленно эскалирует, не дожидаясь лимита.

**Лимит эскалации**: 4 раунда без чистого ревью → Opus пишет краткое описание тупика, пользователь решает: принять / ещё раунд / закрыть.

**Правила авто-коммита**: никогда `--no-verify`, никогда `--amend`. Пропускает авто-коммит, если в дереве есть посторонние незакоммиченные изменения. Никогда не пушит без явного согласия пользователя в текущей сессии.

## Включённые скиллы

Помимо `dev-orchestrator` и `codex-invocation`, плагин содержит 12 скиллов из [superpowers](https://github.com/obra/superpowers) v5.0.7 (без неймспейса):

| Скилл | Что делает |
|---|---|
| `brainstorming` | Интерактивный визуальный брейншторм с компаньоном в браузере |
| `dispatching-parallel-agents` | Паттерн запуска независимых субагентов параллельно |
| `executing-plans` | Руководство по пошаговому выполнению написанного плана |
| `finishing-a-development-branch` | Чеклист перед мержем: тесты, покрытие, ревью, PR |
| `receiving-code-review` | Как обрабатывать и отвечать на замечания код-ревью |
| `requesting-code-review` | Как запросить тщательное код-ревью у субагента |
| `systematic-debugging` | Протокол отладки от корневой причины; антипаттерны под давлением тестов |
| `test-driven-development` | TDD-цикл для Claude Code; справочник антипаттернов |
| `using-git-worktrees` | Безопасная параллельная работа через git worktrees |
| `verification-before-completion` | Чеклист перед тем, как объявить задачу выполненной |
| `writing-plans` | Как писать хороший план реализации |
| `writing-skills` | Как создавать скиллы для Claude Code; лучшие практики Anthropic |

**Код-ревью** диспатчится как Opus-субагент через `Agent(subagent_type="general-purpose", model="opus", ...)` — см. `skills/requesting-code-review/`. Самостоятельный agent type `code-reviewer` из upstream удалён; используйте скилл вместо него.

**Не включено**: `subagent-driven-development` (заменён на `dev-orchestrator`), `using-superpowers` (агрессивное MUST-директивы не нужны), SessionStart-хук.

## Установка

```
/plugin marketplace add github:strigov/superpowers-strigov-ver
/plugin install superpowers-strigov-ver
/reload-plugins
```

### Требования

1. **Claude Code** (любая актуальная версия)
2. **Codex CLI** — `npm install -g @openai/codex` (отдельный CLI; плагин `openai/codex-plugin-cc` в Claude Code устанавливать НЕ нужно — нужный нам runtime вендорится в этот плагин под `vendor/codex-companion/`)
3. **Вход в Codex** — один раз на машину: `!codex login` (префикс `!` запускает команду в вашем шелле)

### Вендоринг Codex companion

Codex companion runtime (родом из `openai/codex-plugin-cc`, Apache-2.0, © OpenAI; см. `vendor/codex-companion/LICENSE` и `NOTICE`) лежит в `vendor/codex-companion/`. Версия фиксируется в `vendor/codex-companion/VERSION`.

Все вызовы Codex идут через обёртку `bin/codex-dispatch`, которая:
- резолвит `vendor/codex-companion/scripts/codex-companion.mjs` либо в marketplace-инсталле (`~/.claude/plugins/cache/strigov-cc-plugins/superpowers-strigov-ver/<version>/...`), либо в локальном dev-дереве;
- пинит `CLAUDE_PLUGIN_DATA` в `~/.claude/plugins/data/superpowers-strigov-ver-codex`, чтобы jobs/broker.sock не пересекались с другими установками codex'а на той же машине;
- для `task`/`review`/`adversarial-review` автоматически инжектит `--model gpt-5.5` (override через env `CODEX_DEFAULT_MODEL=...` или явный `--model`), чтобы бэкенд не делал auto-downgrade на spark на мелких/low-effort задачах.

Чтобы обновить вендорный runtime: скопировать `plugins/codex/{scripts,prompts,schemas}` и top-level `LICENSE`/`NOTICE` из нового релиза `openai/codex-plugin-cc` в `vendor/codex-companion/` и обновить `VERSION`.

## Использование

```
/dev <описание задачи>
```

Или просто опишите задачу на реализацию — `dev-orchestrator` срабатывает на ключевые слова: *реализуй*, *запили*, *накодь*, *напиши*, *implement*, *build*, *write the code* и подобные.

Для тривиальных правок (переименование, опечатка, очевидный однострочный баг) скилл сам направит на Sonnet quickfix — без вызовов Codex.
